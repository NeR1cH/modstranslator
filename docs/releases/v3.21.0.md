## 🎯 Основные изменения

### Исправлена обработка прилагательных с ударным окончанием

Система теперь правильно обрабатывает прилагательные типа "золотой", "большой", "молодой" и корректно согласует их с существительными.

**Примеры:**
- "Gold Sheet" → "Золотой лист" ✅ (было: "Золотый лист" ❌)
- "Gold Block" → "Золотой блок" ✅ (было: "Золотый блок" ❌)
- "Big Sword" → "Большой меч" ✅
- "Young Dragon" → "Молодой дракон" ✅

## 🐛 Исправленные баги

### 1. Неправильная нормализация прилагательных

**Проблема:**
- `normalizeToMasculine()` возвращала "Золотой" вместо нормализации к базовой форме "Золотый"
- При сохранении в кеш терялась информация об ударном окончании

**Решение:**
- Функция теперь всегда нормализует к -ый/-ий форме
- Флаг `stressedEnding = true` сохраняется отдельно
- При применении согласования система использует этот флаг для выбора правильного окончания

**Код:**
```typescript
// Было:
if (stem.endsWith('ой')) {
  return [stem, true]; // Возвращала "Золотой" как есть
}

// Стало:
if (stem.endsWith('ой')) {
  stem = stem.slice(0, -2);
  stressedEnding = true;
}
// ... затем нормализует к "Золотый" с флагом stressedEnding = true
```

### 2. stressedEnding не обновлялся при повторном обучении

**Проблема:**
- Когда система учила "Gold Ore" → "Золотая руда", сохранялся `stressedEnding = false`
- Потом при обучении "Gold Ingot" → "Золотой слиток" флаг не обновлялся на `true`
- Результат: система не знала, что "Gold" требует ударного окончания

**Решение:**
- При обновлении фрагмента `stressedEnding = true` теперь перезаписывает `false`
- Логика: если хотя бы раз встретилось ударное окончание, оно приоритетнее

**Код:**
```typescript
// Было:
if (stressedEnding !== undefined && existing.stressedEnding === undefined) {
  existing.stressedEnding = stressedEnding;
}

// Стало:
if (stressedEnding !== undefined) {
  if (existing.stressedEnding === undefined || stressedEnding === true) {
    existing.stressedEnding = stressedEnding;
  }
}
```

### 3. Падающий тест fragmentCacheVariation.test.ts

**Проблема:**
- Тест пытался делать реальные API вызовы к OpenRouter
- `global.fetch = jest.fn()` был недостаточен — нужен был полный мок

**Решение:**
- Добавлен мок, который возвращает правильные переводы для каждого текста
- Мок поддерживает батчи через разделитель `###SPLIT###`
- Все переводы теперь детерминированы и не требуют API ключа

**Код:**
```typescript
global.fetch = jest.fn((url, options) => {
  const body = JSON.parse((options as any)?.body || '{}');
  const userMessage = body.messages?.find((m: any) => m.role === 'user')?.content || '';

  // Handle batch translation (texts joined with ###SPLIT###)
  if (userMessage.includes('###SPLIT###')) {
    const texts = userMessage.split('\n###SPLIT###\n');
    const translations = texts.map((text: string) =>
      openRouterTranslations[text.trim()] || `UNMOCKED: ${text}`
    );
    return Promise.resolve({
      ok: true,
      status: 200,
      json: async () => ({
        choices: [{ message: { content: translations.join('\n###SPLIT###\n') } }]
      })
    } as Response);
  }
  // ...
});
```

### 4. Case-sensitive regex в fragmentCache.grammar.test.ts

**Проблема:**
- Тест ожидал "ый...ый слиток", но получал "Очищенный стальной слиток"
- Regex не учитывала заглавную букву в начале предложения

**Решение:**
- Добавлен флаг `/i` для всех regex проверок

**Код:**
```typescript
// Было:
expect(result3).toMatch(/ый.*ый слиток|ой.*ой слиток/);

// Стало:
expect(result3).toMatch(/ый.*ый слиток|ой.*ой слиток/i);
```
