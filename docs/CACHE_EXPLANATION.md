#### 2. Исправление BaseCache инициализации (v3.15.0)
**Проблема:** При запуске сервера ошибка "Cannot read properties of undefined (reading 'set')".

**Причина:** BaseCache вызывал `init()` → `loadFromDisk()` до того, как TranslationCache инициализировал `memoryCache`.

**Решение:**
- Изменен `init()` с `private` на `protected`
- Убран автоматический вызов `init()` из конструктора BaseCache
- TranslationCache теперь явно вызывает `this.init()` после `super()`

**Результат:** Сервер запускается без ошибок ✅

---

#### 3. Исправление Translation Cache (v3.15.1) ⭐ ГЛАВНОЕ
**Проблема:** Fragment cache не рос, оставался на уровне 138 фрагментов, несмотря на 12,560 переводов в кэше.

**Причина:** Translation cache хранил `original: ''` (пустую строку) для экономии места. Fragment cache не мог извлекать паттерны без оригинального текста.

**Решение:** Изменена структура memoryCache:

```typescript
// БЫЛО (v3.14.0 и раньше):
private memoryCache = new Map<string, string>();
// Хранили только перевод: { hash → "Алмазный меч" }

// СТАЛО (v3.15.1):
private memoryCache = new Map<string, { original: string; translated: string }>();
// Храним пару: { hash → { original: "Diamond Sword", translated: "Алмазный меч" } }
```

**Обновлены методы:**
- `get()` - возвращает `cached.translated`
- `set()` - сохраняет `{ original, translated }`
- `setMany()` - сохраняет пары
- `loadFromDisk()` - загружает структуру
- `saveToDisk()` - сохраняет `original` вместо `''`

**Результат:**
- ✅ Новые переводы сохраняют оригинальный текст
- ✅ Fragment cache сможет извлекать паттерны из будущих переводов
- ✅ Ожидаемый рост: 138 → 2,000-3,000 фрагментов после следующего перевода
- ✅ Размер cache файла +30% (приемлемо)

## 🔍 Как работает новый кэш (v3.15.1)

### Translation Cache

**Файл:** `lib/translationCache.ts`  
**Хранилище:** `.translation-cache/cache-v1.json`

**Структура данных:**
```typescript
{
  version: "v1",
  entries: [
    {
      hash: "abc123...",           // SHA-256 хэш от (original + targetLang)
      original: "Diamond Sword",   // ⭐ НОВОЕ: оригинальный текст
      translated: "Алмазный меч",  // переведенный текст
      timestamp: 1778251156650     // время создания
    }
  ]
}
```

**Как работает:**
1. Приходит текст для перевода: `"Diamond Sword"`
2. Вычисляется хэш: `hash("Diamond Sword" + "RU")`
3. Проверяется кэш: `memoryCache.get(hash)`
   - **HIT** → возвращаем `cached.translated` ("Алмазный меч")
   - **MISS** → вызываем DeepL API
4. Сохраняем результат: `memoryCache.set(hash, { original: "Diamond Sword", translated: "Алмазный меч" })`
5. Автосохранение на диск каждые 5 секунд

**Важно:** Теперь кэш хранит **оригинал + перевод**, что позволяет fragment cache извлекать паттерны!

---

### Fragment Cache

**Файл:** `lib/fragmentCache.ts`  
**Хранилище:** `.translation-cache/fragments-v1.json`

**Структура данных:**
```typescript
{
  version: "v1",
  fragments: {
    "diamond": {
      text: "diamond",
      translation: "Алмазный",
      context: "prefix",
      count: 15,              // сколько раз встречался
      confidence: 95,         // уверенность (50-100%)
      gender: "masculine"     // ⭐ НОВОЕ: род существительного
    },
    "sword": {
      text: "sword",
      translation: "меч",
      context: "suffix",
      count: 23,
      confidence: 98,
      gender: "masculine"
    }
  }
}
```

**Как работает:**

#### Шаг 1: Обучение (после DeepL перевода)
```typescript
// Получили перевод от DeepL:
original: "Diamond Sword"
translated: "Алмазный меч"

// Извлекаем паттерны:
extractPatterns("Diamond Sword", "Алмазный меч")
  → Pattern 1: "Material + Item" (2 слова)
  → Материал: "diamond" → "Алмазный" (мужской род)
  → Предмет: "sword" → "меч" (мужской род)

// Сохраняем фрагменты:
fragments.set("diamond", { translation: "Алмазный", gender: "masculine", ... })
fragments.set("sword", { translation: "меч", gender: "masculine", ... })
```

#### Шаг 2: Использование (при следующем переводе)
```typescript
// Приходит новый текст:
"Iron Sword"

// Проверяем Translation Cache:
translationCache.get("Iron Sword") → MISS

// Проверяем Fragment Cache:
tryTranslate("Iron Sword")
  → Распознаем паттерн: "Material + Item"
  → Ищем фрагменты: "iron" → "Железный", "sword" → "меч"
  → Применяем согласование: applyGenderAgreement("Железный", "masculine")
  → Собираем: "Железный меч" ✅

// Возвращаем перевод БЕЗ вызова DeepL API!
```

**Поддерживаемые паттерны:**

1. **Pattern 1: Material + Item** (2 слова)
   - `"Diamond Sword"` → `"Алмазный меч"`
   - `"Iron Ore"` → `"Железная руда"`

2. **Pattern 2: Single Word** (1 слово)
   - `"Diamond"` → `"Алмаз"`
   - `"Sword"` → `"Меч"`

3. **Pattern 3: Prefix + Material + Item** (3 слова)
   - `"Raw Iron Ore"` → `"Сырая железная руда"`
   - `"Crushed Copper Dust"` → `"Измельченная медная пыль"`

**Словари:**
- `MATERIALS` - 41 материал (diamond, iron, gold, copper, steel, ...)
- `ITEM_TYPES` - 43 типа предметов (sword, ore, ingot, dust, block, ...)
- `PREFIXES` - 10 префиксов (raw, crushed, refined, ...)
- `ITEM_GENDERS` - 63 слова с родами (sword: masculine, ore: feminine, ...)

## 🎯 Грамматическое согласование

**Проблема:** Русский язык требует согласования прилагательных с существительными по роду.

**Пример:**
```typescript
// Без согласования:
"Lead" (свинцовый) + "Nugget" (самородок, м.р.) → "Свинцовая самородок" ❌

// С согласованием:
applyGenderAgreement("Свинцовая", "masculine")
  → stem: "Свинцов" (удалить -ая)
  → ending: "ый" (мужской род)
  → result: "Свинцовый" ✅
  
Итог: "Свинцовый самородок" ✅
```

**Правила окончаний:**
```typescript
masculine:   -ый, -ий, -ой  (Алмазный, Синий, Золотой)
feminine:    -ая, -яя       (Железная, Синяя)
neuter:      -ое, -ее       (Золотое, Синее)
```

**Функция:** `applyGenderAgreement(adjective: string, gender: string)`

**Точность:** 85-90% (не обрабатываются исключения типа "золотой/стальной")

## 📝 Правила работы с кэшем

### ✅ DO (Делать)

1. **Всегда сохранять оригинальный текст** в Translation Cache
2. **Извлекать фрагменты после каждого DeepL перевода** (автоматически)
3. **Применять грамматическое согласование** при сборке из фрагментов
4. **Проверять Fragment Cache перед DeepL API** (экономия вызовов)
5. **Автосохранение кэша каждые 5 секунд** (защита от потери данных)

### ❌ DON'T (Не делать)

1. **Не удалять оригинальный текст** из Translation Cache (даже для экономии места)
2. **Не пытаться извлечь фрагменты без оригинала** (не работает)
3. **Не игнорировать род существительных** (приведет к грамматическим ошибкам)
4. **Не вызывать DeepL API если есть фрагменты** (лишние расходы)
5. **Не изменять структуру кэша без обновления версии** (breaking change)

## 🔧 Технические детали

### Архитектура кэширования

```
BaseCache (abstract)
├── TranslationCache
│   ├── memoryCache: Map<hash, {original, translated}>  ⭐ НОВОЕ
│   ├── cacheFile: .translation-cache/cache-v1.json
│   └── autoSave: 5 seconds
│
└── FragmentCache
    ├── fragments: Map<key, Fragment>
    ├── cacheFile: .translation-cache/fragments-v1.json
    ├── MATERIALS: 41 items
    ├── ITEM_TYPES: 43 items
    ├── PREFIXES: 10 items
    └── ITEM_GENDERS: 63 items  ⭐ НОВОЕ
```

### Процесс перевода с фрагментами

```
1. Получить текст для перевода: "Iron Sword"
2. Проверить Translation Cache (hash lookup)
   └─ HIT → вернуть перевод
3. Проверить Fragment Cache (pattern detection)
   └─ HIT → собрать из фрагментов + применить согласование
4. Вызвать DeepL API
5. Сохранить в Translation Cache (с original текстом) ⭐ ВАЖНО
6. Извлечь фрагменты и сохранить в Fragment Cache (с родом)
```
